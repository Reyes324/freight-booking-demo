# 地址搜索模块

## 交互设计说明

### 用户场景
**谁**：需要下单叫货运车辆的企业用户或个人用户

**场景**：在货运下单页面填写装货/卸货地址，用户需要快速、准确地输入地址并补充联系人信息

**为什么需要这个模块**：
- 地址是货运订单的核心信息，必须准确无误
- 手动输入地址容易出错且效率低，需要搜索联想提升体验
- 需要收集收/发货人联系方式（姓名、电话、门牌号），但不能阻断 Demo 演示流程

### 操作流程

```
用户进入下单页面
    ↓
点击"装货地址"或"卸货地址"输入框
    ↓
输入 >= 1 个字符
    ↓
200ms debounce 后触发 API 搜索
    ↓
显示加载状态（蓝色脉冲点）
    ↓
┌─────────────────────────────────────┐
│ 有结果                               │ 无结果（输入 < 3 字符）
│  ↓                                   │  ↓
│ 显示下拉列表（PC）或底部抽屉（移动） │ 隐藏下拉/抽屉
│  ↓                                   │
│ 用户点击某个地址建议                 │ 无结果（输入 >= 3 字符）
│  ↓                                   │  ↓
│ 输入框填充选中地址主文本             │ 显示空状态提示
│  ↓                                   │  ↓
│ 弹出地址详情模态窗                   │ 用户继续输入或修改关键词
│  ↓                                   │  ↓
│ 用户填写收货人姓名、电话、门牌号     │ 回到搜索流程
│ （可选字段，无验证）                 │
│  ↓                                   │
│ 点击"确认"按钮                       │
│  ↓                                   │
│ 模态窗关闭                           │
│  ↓                                   │
│ 路线区域显示已确认的完整地址信息     │
│  ↓                                   │
│ 用户点击"编辑"按钮（如需修改）       │
│  ↓                                   │
│ 重新打开模态窗，回填已有数据         │
│  ↓                                   │
│ 修改后点击"确认"，更新地址           │
└─────────────────────────────────────┘

用户点击模态窗外部或"取消"按钮
    ↓
模态窗关闭，输入框清空，不保存地址
```

### UI 状态清单

| 状态名称 | 触发条件 | 视觉表现 |
|---------|---------|---------|
| 默认态 | 未选择地址，输入框为空 | 灰色边框输入框，placeholder "装货地址"/"卸货地址" |
| 输入中 | 用户正在输入 | 蓝色边框 + 蓝色光晕（ring），无图标 |
| 搜索加载中 | 输入停止后 200ms，API 请求中 | 输入框右侧显示蓝色脉冲点动画 |
| 有结果 - 下拉显示 (PC) | 输入 >= 1 字符且 API 返回结果 > 0 | 输入框下方浮动白色圆角卡片，显示建议列表，z-index: 50 |
| 有结果 - 底部抽屉显示 (移动) | 同上 | 屏幕底部向上弹出抽屉（70vh），半透明遮罩，z-index: 40/50 |
| 无结果 - 短关键词 | 输入 1-2 字符且 API 无结果 | 不显示下拉/抽屉 |
| 无结果 - 长关键词 | 输入 >= 3 字符且 API 无结果 | 显示空状态：搜索图标 + "未找到匹配的地址" + "请尝试其他关键词" |
| 悬停建议项 (PC) | 鼠标悬停在某个地址建议上 | 背景变为浅灰色 bg-gray-50 |
| 点击建议项 (移动) | 手指按下建议项 | 背景变为 bg-gray-100 |
| 已选择地址 | 用户确认地址详情 | 输入框消失，显示完整地址信息卡片：主地址 + 联系人 + 电话 + 门牌号，右侧"编辑"按钮 |
| 模态窗打开 | 点击地址建议或"编辑"按钮 | 半透明黑色遮罩 (bg-black/40)，居中白色圆角卡片 (PC) 或底部抽屉 (移动)，z-index: 100/110 |
| 模态窗关闭动画 | 点击取消/确认/遮罩 | fade-out + slide-out 动画 200ms (PC) / 300ms (移动) |
| 编辑模式 | 点击"编辑"按钮 | 重新打开模态窗，所有字段回填已有数据 |

### 设计决策与理由

#### 1. 搜索触发阈值：1 个字符即触发
**决策来源**：`design-decisions.md` - 地址搜索模块 - 搜索触发阈值

**理由**：用户输入地址时通常知道自己要搜什么，1 个字符就开始联想能让用户更快找到目标，减少等待感。原来设置的 2 字符门槛增加了不必要的摩擦。

**代码实现**：`AddressSearchInput.tsx` 第 42 行
```typescript
if (value.length > 0) {
  setIsLoading(true);
  searchTimeoutRef.current = setTimeout(async () => {
    // 触发搜索
  }, 200);
}
```

#### 2. Debounce 时间：200ms
**决策来源**：`design-decisions.md` - 地址搜索模块 - Debounce 时间

**理由**：从 500ms 降到 200ms，追求即时响应感。500ms 的延迟让人感觉系统在"卡"，200ms 接近即时反馈，同时还能避免过于频繁的 API 调用。

**代码实现**：`AddressSearchInput.tsx` 第 57 行
```typescript
setTimeout(async () => {
  // 搜索逻辑
}, 200);
```

#### 3. 表单验证策略：所有字段可选
**决策来源**：`design-decisions.md` - 地址搜索模块 - 表单验证策略

**理由**：这是 Demo 演示场景，不是生产环境。强制填写会阻断演示流程，让演示变得啰嗦。用户（演示者）应该能快速走完整个下单流程，不被表单验证卡住。

**代码实现**：`AddressDetailsModal.tsx` 第 34-40 行
```typescript
const handleSubmit = () => {
  onConfirm({
    address: addressText,
    contactName: contactName.trim() || undefined,
    phone: phone.trim() || undefined,
    unit: unit.trim() || undefined,
  });
};
```
确认按钮始终可点击，无禁用状态。

#### 4. 地址详情弹窗设计：模态窗 vs 同页展开
**决策来源**：`design-decisions.md` - 地址搜索模块 - 地址详情弹窗设计

**理由**：模态窗能聚焦用户注意力到当前任务（填写收货信息），避免页面其他元素干扰。同时模态窗的"确认/取消"明确了操作的完成边界。

**视觉表现**：
- PC：屏幕居中，最大宽度 448px，圆角 2xl，阴影 2xl
- 移动：底部抽屉，最大高度 85vh，圆角顶部 2xl

#### 5. PC 端下拉 vs 移动端底部抽屉
**决策来源**：`design-decisions.md` - 地址搜索模块 - PC 端下拉 vs 移动端底部抽屉

**理由**：
- PC：鼠标操作，下拉列表在输入框附近是最自然的位置，减少视线移动
- 移动端：拇指可达性是关键，底部抽屉在屏幕下半区，单手操作更舒适。这也是移动端原生应用的主流交互模式

**代码实现**：
- PC：`AddressSearchInput.tsx` 第 128-196 行，`hidden lg:block`
- 移动：`AddressSearchInput.tsx` 第 198-280 行，`lg:hidden`

#### 6. 空状态显示阈值：>= 3 字符
**决策来源**：`design-decisions.md` - 地址搜索模块 - 空状态显示阈值

**理由**：用户只输入 1-2 个字符时，搜索结果为空是正常的（关键词太短）。此时显示"未找到"会造成误导，让用户以为地址不存在。3 个字符以上无结果，才值得提示用户。

**代码实现**：`AddressSearchInput.tsx` 第 48-49、53 行
```typescript
// 有结果时显示,或者输入>=3个字符但没结果时也显示空状态
setShowDropdown(results.length > 0 || value.length >= 3);
```

#### 7. 已选地址的编辑模式
**决策来源**：`design-decisions.md` - 地址搜索模块 - 已选地址的编辑模式

**理由**：用户可能选错地址或需要修改联系人信息，编辑功能让操作可逆。回填已有数据避免用户重复输入。

**代码实现**：`RouteSection.tsx` 第 75-83 行（装货地址编辑按钮）
```typescript
<button
  onClick={() => {
    setSelectedAddressText(pickupAddress.address);
    setShowPickupModal(true);
  }}
  className="flex-shrink-0 text-xs text-blue-600 hover:text-blue-700 font-medium"
>
  编辑
</button>
```

模态窗通过 `initialData` prop 回填数据：`AddressDetailsModal.tsx` 第 25-32 行

#### 8. 地图 API 选择：高德地图
**决策来源**：`design-decisions.md` - 地址搜索模块 - 地图 API 选择

**理由**：从 Google Maps 切换到高德地图。Google Maps API 存在结算问题（需要绑定信用卡），高德地图在香港地区同样有良好覆盖，且接入更简单。

**技术影响**：创建了 Service 抽象层 (`addressService.ts`)，可以通过换一行 import 切换不同的地图服务。

**当前实现**：`addressService.ts` 第 18-20 行
```typescript
// Current implementation - Gaode Maps API
import { searchAddressGaode } from './gaode';
export const searchAddress = searchAddressGaode;
```

#### 9. API Key 安全
**决策来源**：`design-decisions.md` - 地址搜索模块 - API Key 安全

**理由**：即使是 Demo，也应该遵循基本的安全实践。API Key 只存在于服务端。

**代码实现**：通过 Next.js API Route 代理外部 API 调用
- 前端调用：`gaode.ts` 第 12-14 行，调用 `/api/search-gaode`
- 服务端代理：`app/api/search-gaode/route.ts`，API Key 存储在第 3 行

#### 10. UI 层级管理：z-index 策略
**决策来源**：`design-decisions.md` - UI 层级管理 - z-index 策略

**理由**：解决实际开发中遇到的 bug — 搜索下拉和地址详情弹窗会同时出现，z-index 冲突导致弹窗被下拉遮挡。通过拉大层级差距确保弹窗始终在最上层。

**层级分配**：
- 搜索下拉（PC）：z-50 (`AddressSearchInput.tsx` 第 131 行)
- 搜索抽屉遮罩（移动）：z-40 (第 202 行)
- 搜索抽屉主体（移动）：z-50 (第 205 行)
- 模态窗遮罩：z-[100] (`AddressDetailsModal.tsx` 第 49 行)
- 模态窗主体：z-[110] (第 54、158 行)

#### 11. 防止选择后二次搜索：justSelectedRef
**决策来源**：`design-decisions.md` - UI 层级管理 - 防止选择后二次搜索

**理由**：用户从建议列表选择地址后，选中的地名会填入输入框，触发 value 变化 → 触发搜索。这不是用户意图，会导致刚关闭的下拉又弹出来。这是实际开发中发现的 bug，通过 Ref（而非 State）标志位解决，因为需要同步读写且不需要触发重渲染。

**代码实现**：`AddressSearchInput.tsx`
- 定义：第 28 行 `const justSelectedRef = useRef(false);`
- 检测：第 36-39 行，如果刚选择过地址，不触发搜索
- 设置：第 87 行，选择地址时标记为 true

## 组件详情

### AddressSearchInput

**文件**：`src/components/AddressSearchInput.tsx`

**功能**：地址搜索输入框，输入关键词后显示建议列表（PC 端下拉卡片，移动端底部抽屉）。

#### Props

| 名称 | 类型 | 必填 | 默认值 | 描述 |
|-----|------|------|-------|------|
| value | string | 是 | - | 输入框当前值（受控组件） |
| onChange | (value: string) => void | 是 | - | 输入值变化回调 |
| onSelectAddress | (address: AddressSuggestion) => void | 是 | - | 用户选择某个地址建议时的回调 |
| placeholder | string | 否 | "搜索地址" | 输入框占位符 |
| label | string | 否 | undefined | 输入框上方的标签文本 |

#### 内部状态

| 状态名称 | 类型 | 初始值 | 用途 |
|---------|------|-------|------|
| showDropdown | boolean | false | 控制建议列表的显示/隐藏 |
| suggestions | AddressSuggestion[] | [] | 存储 API 返回的地址建议列表 |
| isLoading | boolean | false | 标记是否正在进行 API 请求 |

#### Ref 变量

| Ref 名称 | 类型 | 用途 |
|---------|------|------|
| inputRef | HTMLInputElement | 引用输入框 DOM 元素，用于点击外部检测 |
| dropdownRef | HTMLDivElement | 引用下拉列表 DOM 元素，用于点击外部检测 |
| searchTimeoutRef | NodeJS.Timeout | 存储 debounce 定时器 ID，用于清理 |
| justSelectedRef | boolean | 标记用户是否刚从建议列表选择了地址，防止二次触发搜索 |

#### 用户交互

| 交互元素 | 交互行为 | 触发的逻辑 |
|---------|---------|-----------|
| 输入框 | 输入文本 | onChange 回调 → 触发 useEffect → 200ms debounce → 调用 searchAddress API → 更新 suggestions 和 showDropdown |
| 输入框 | 获得焦点 | 蓝色边框 + ring 动画（CSS） |
| 建议列表项 (PC) | 鼠标悬停 | 背景色变为 bg-gray-50 |
| 建议列表项 | 点击 | handleSelect → 设置 justSelectedRef = true → onChange(地址主文本) → onSelectAddress(完整地址对象) → 关闭下拉/抽屉 |
| 下拉列表外部区域 | 鼠标按下 (mousedown) | handleClickOutside → 关闭下拉 |
| 底部抽屉遮罩 | 点击 | 关闭抽屉 |

#### 状态变化流程图

```
[输入框为空]
    ↓ 用户输入字符
[value.length > 0] → setIsLoading(true)
    ↓ 200ms debounce
[调用 searchAddress API]
    ↓
    ├─ API 返回结果 > 0
    │   ↓
    │  setSuggestions(results)
    │  setShowDropdown(true)
    │  setIsLoading(false)
    │   ↓
    │  [显示下拉/抽屉]
    │   ↓ 用户点击某个建议
    │  handleSelect
    │   ↓
    │  justSelectedRef = true
    │  onChange(mainText)
    │  onSelectAddress(suggestion)
    │  setShowDropdown(false)
    │   ↓
    │  [关闭下拉/抽屉]
    │
    ├─ API 返回结果 = 0 且 value.length >= 3
    │   ↓
    │  setSuggestions([])
    │  setShowDropdown(true)
    │  setIsLoading(false)
    │   ↓
    │  [显示空状态]
    │
    └─ API 返回结果 = 0 且 value.length < 3
        ↓
       setSuggestions([])
       setShowDropdown(false)
       setIsLoading(false)
        ↓
       [不显示下拉/抽屉]
```

#### 响应式行为

| 断点 | 显示方式 | 类名控制 |
|-----|---------|---------|
| < lg (< 1024px) | 底部抽屉 + 遮罩 | `lg:hidden` |
| >= lg | 输入框下方浮动卡片 | `hidden lg:block` |

### AddressDetailsModal

**文件**：`src/components/AddressDetailsModal.tsx`

**功能**：地址详情填写模态窗，让用户填写收货人姓名、电话、门牌号。

#### Props

| 名称 | 类型 | 必填 | 默认值 | 描述 |
|-----|------|------|-------|------|
| isOpen | boolean | 是 | - | 控制模态窗显示/隐藏 |

## 关键实现细节

### 1. Debounce 实现

**位置**：`AddressSearchInput.tsx` 第 30-68 行

**机制**：使用 `useEffect` + `setTimeout` + `useRef` 存储 timeout ID

**关键点**：
- 每次 `value` 变化时清除上一个定时器
- 200ms 后触发搜索
- 组件卸载时清理定时器（useEffect 返回清理函数）

```typescript
useEffect(() => {
  if (searchTimeoutRef.current) {
    clearTimeout(searchTimeoutRef.current);
  }

  if (value.length > 0) {
    setIsLoading(true);
    searchTimeoutRef.current = setTimeout(async () => {
      // 搜索逻辑
    }, 200);
  }

  return () => {
    if (searchTimeoutRef.current) {
      clearTimeout(searchTimeoutRef.current);
    }
  };
}, [value]);
```

### 2. justSelectedRef 防止二次搜索

**位置**：`AddressSearchInput.tsx` 第 28、36-39、87 行

**问题**：用户点击建议列表某个地址后，地址文本会填入输入框（触发 `onChange`），导致 `value` 变化 → `useEffect` 触发 → 再次搜索 → 下拉列表重新打开。这不是用户意图。

**解决方案**：使用 `useRef` 存储布尔标志位
- 点击建议时设置 `justSelectedRef.current = true`
- `useEffect` 中检测到此标志位时跳过搜索
- 跳过后立即重置为 `false`

**为什么用 Ref 而非 State？**
- Ref 的读写是同步的，立即生效
- State 的更新是异步的，可能导致时序问题
- Ref 的变化不会触发重渲染，避免不必要的性能开销

```typescript
const justSelectedRef = useRef(false);

useEffect(() => {
  if (justSelectedRef.current) {
    justSelectedRef.current = false;
    return; // 跳过搜索
  }
  // 正常搜索逻辑
}, [value]);

const handleSelect = (suggestion: AddressSuggestion) => {
  justSelectedRef.current = true;
  onChange(suggestion.mainText);
  onSelectAddress(suggestion);
  setShowDropdown(false);
};
```

### 3. 点击外部关闭：mousedown vs click

**位置**：`AddressSearchInput.tsx` 第 71-84 行

**为什么用 `mousedown` 而非 `click`？**

因为建议列表项的 `onClick` 事件触发时，事件冒泡会传播到 document。如果用 `click` 监听，会出现：
1. 用户点击建议
2. 建议的 `onClick` 触发，关闭下拉
3. 事件冒泡到 document，`handleClickOutside` 再次触发，可能干扰状态

`mousedown` 事件在 `click` 之前触发，时序更清晰。

```typescript
useEffect(() => {
  function handleClickOutside(event: MouseEvent) {
    if (
      dropdownRef.current &&
      !dropdownRef.current.contains(event.target as Node) &&
      !inputRef.current?.contains(event.target as Node)
    ) {
      setShowDropdown(false);
    }
  }

  document.addEventListener("mousedown", handleClickOutside);
  return () => document.removeEventListener("mousedown", handleClickOutside);
}, []);
```

### 4. z-index 层级管理

**问题**：搜索下拉列表和地址详情模态窗可能同时出现（例如用户点击建议后，下拉列表尚未完全消失，模态窗已经开始渲染）。如果层级设置不当，会导致模态窗被下拉列表遮挡。

**解决方案**：拉大层级差距

| 元素 | z-index | 文件位置 |
|-----|---------|---------|
| 搜索下拉（PC） | 50 | `AddressSearchInput.tsx` 第 131 行 |
| 搜索抽屉遮罩（移动） | 40 | 第 202 行 |
| 搜索抽屉主体（移动） | 50 | 第 205 行 |
| 模态窗遮罩 | 100 | `AddressDetailsModal.tsx` 第 49 行 |
| 模态窗主体（PC） | 110 | 第 54 行 |
| 模态窗主体（移动） | 110 | 第 158 行 |

这样即使下拉和模态窗同时存在，模态窗也始终在最上层。

### 5. 模态窗数据回填

**位置**：`AddressDetailsModal.tsx` 第 25-32 行

**机制**：通过 `useEffect` 监听 `isOpen` 和 `initialData`，当模态窗打开时重新初始化表单数据

**为什么需要 useEffect？**

因为 `initialData` 可能在组件挂载后才传入（编辑模式下，用户点击"编辑"按钮时才设置 `initialData`）。如果只在 `useState` 的初始值中读取 `initialData`，会导致编辑模式下数据无法回填。

```typescript
const [contactName, setContactName] = useState(initialData?.contactName || "");
const [phone, setPhone] = useState(initialData?.phone || "");
const [unit, setUnit] = useState(initialData?.unit || "");

useEffect(() => {
  if (isOpen) {
    setContactName(initialData?.contactName || "");
    setPhone(initialData?.phone || "");
    setUnit(initialData?.unit || "");
  }
}, [isOpen, initialData]);
```

### 6. 可选字段的数据处理

**位置**：`AddressDetailsModal.tsx` 第 34-40 行

**策略**：空字符串转为 `undefined`，避免传递无意义的空值

```typescript
const handleSubmit = () => {
  onConfirm({
    address: addressText,
    contactName: contactName.trim() || undefined,
    phone: phone.trim() || undefined,
    unit: unit.trim() || undefined,
  });
};
```

**为什么这样做？**
- TypeScript 接口中字段类型为 `string | undefined`，不包括空字符串
- 空字符串和 `undefined` 语义不同：空字符串表示"用户填了但没填内容"，`undefined` 表示"用户没填"
- 后续逻辑中可以用 `address.contactName && ...` 安全检测（不会误判空字符串为 truthy）

### 10. 响应式断点：lg (1024px)

**全局断点**：Tailwind CSS 默认断点
- `sm`: 640px
- `md`: 768px
- `lg`: 1024px ← 本项目使用
- `xl`: 1280px
- `2xl`: 1536px

**选择 lg 的原因**：
- iPad Pro (1024px) 及以上设备使用桌面布局
- 小于 1024px 的设备（手机、小平板）使用移动布局
- 这是现代 Web 应用的主流分界点

**实现方式**：Tailwind 的移动优先（Mobile First）策略
```html
<!-- 默认（移动端）：底部抽屉 -->
<div className="lg:hidden fixed bottom-0 ...">
  <!-- 移动端内容 -->
</div>

<!-- lg 及以上（桌面端）：浮动卡片 -->
<div className="hidden lg:block absolute ...">
  <!-- 桌面端内容 -->
</div>
```
