# 设计决策记录

> 本文件记录交互设计师在开发过程中的设计决策及其理由。
> 由主对话中的 Claude 在与设计师沟通后整理，供 doc-writer 生成模块文档时引用。
>
> 格式：每条决策包含「做了什么」和「为什么」。

## 全局决策

### 项目定位
- **决策**：这是一个高保真可交互 Demo，不是生产代码
- **理由**：给老板和技术团队演示用。需要看起来真实、可操作，但不需要完整的后端逻辑
- **影响**：优先保证交互完整性和视觉效果，数据可以 mock，接口可以简化

### 开发节奏
- **决策**：逐步迭代 — 一个页面 → 一个模块 → 叠加扩展
- **理由**：先出第一版让团队看到整体框架，再逐模块打磨细节，最后统一适配移动端
- **影响**：每个模块需要独立可用，不依赖未完成的模块

### 响应式策略
- **决策**：先做 PC 版，移动端后续统一处理
- **理由**：demo 演示主要在 PC 上，移动端适配是锦上添花
- **影响**：组件预留了响应式断点（lg:），但移动端细节暂不打磨

### 地区与语言
- **决策**：面向香港地区，UI 使用繁体中文 + 英文混合
- **理由**：产品定位是企业国际版，首发市场是香港

## 地址搜索模块

### 搜索触发阈值
- **决策**：1 个字符即触发搜索联想
- **理由**：用户输入地址时通常知道自己要搜什么，1 个字符就开始联想能让用户更快找到目标，减少等待感。原来设置的 2 字符门槛增加了不必要的摩擦

### Debounce 时间
- **决策**：从 500ms 降到 200ms
- **理由**：追求即时响应感。500ms 的延迟让人感觉系统在"卡"，200ms 接近即时反馈，同时还能避免过于频繁的 API 调用

### 表单验证策略
- **决策**：移除所有必填验证，所有字段（收货人、电话、门牌号）改为可选，确认按钮常亮
- **理由**：这是 Demo 演示场景，不是生产环境。强制填写会阻断演示流程，让演示变得啰嗦。用户（演示者）应该能快速走完整个下单流程，不被表单验证卡住

### 地址详情弹窗设计
- **决策**：选择地址后弹出模态窗填写联系人信息，而不是在同一页面内展开
- **理由**：模态窗能聚焦用户注意力到当前任务（填写收货信息），避免页面其他元素干扰。同时模态窗的"确认/取消"明确了操作的完成边界

### PC 端下拉 vs 移动端底部抽屉
- **决策**：PC 端搜索建议用输入框下方的浮动下拉卡片，移动端用底部抽屉
- **理由**：
  - PC：鼠标操作，下拉列表在输入框附近是最自然的位置，减少视线移动
  - 移动端：拇指可达性是关键，底部抽屉在屏幕下半区，单手操作更舒适。这也是移动端原生应用的主流交互模式

### 空状态显示阈值
- **决策**：输入 >= 3 字符且无结果时才显示"未找到"，1-2 字符无结果时不显示
- **理由**：用户只输入 1-2 个字符时，搜索结果为空是正常的（关键词太短）。此时显示"未找到"会造成误导，让用户以为地址不存在。3 个字符以上无结果，才值得提示用户

### 已选地址的编辑模式
- **决策**：地址确认后显示完整信息 + 编辑按钮，点击编辑重新打开弹窗并回填数据
- **理由**：用户可能选错地址或需要修改联系人信息，编辑功能让操作可逆。回填已有数据避免用户重复输入

### 地图 API 选择
- **决策**：从 Google Maps 切换到高德地图
- **理由**：Google Maps API 存在结算问题（需要绑定信用卡），高德地图在香港地区同样有良好覆盖，且接入更简单
- **技术影响**：创建了 Service 抽象层（addressService.ts），可以通过换一行 import 切换不同的地图服务

### API Key 安全
- **决策**：通过 Next.js API Route 代理外部 API 调用，不在前端暴露 API Key
- **理由**：即使是 Demo，也应该遵循基本的安全实践。API Key 只存在于服务端

## 车型选择模块

### 车型卡片交互
- **决策**：点击选中，蓝色边框 + 右上角勾选标记，选中后展开详情
- **理由**：蓝色边框提供清晰的选中反馈，勾选标记是"已选择"的通用视觉语言。展开详情让用户在选择前了解车型规格，无需跳转

### 横向滚动 + 滚动箭头
- **决策**：车型卡片超出容器宽度时可横向滚动，右侧有箭头按钮
- **理由**：保持单行布局的整洁感，同时通过箭头按钮提示用户还有更多选项

### 切换车型重置服务选项
- **决策**：切换车型时，已选的服务选项清空
- **理由**：不同车型对应不同的服务和价格，保留旧的选择会造成数据不一致

## 服务选项 + 价格模块

### 条件显示逻辑
- **决策**：服务选项区域仅当「装货地址 + 卸货地址 + 车型」三个条件同时满足时才显示
- **理由**：没有完整信息时显示价格是无意义的，会让用户困惑。渐进式展示引导用户按步骤完成操作

### "下一步"按钮
- **决策**：按钮已显示但暂不实现跳转
- **理由**：Demo 第一版先聚焦下单页面的完整交互，下一步的流程后续再迭代

## UI 层级管理

### z-index 策略
- **决策**：搜索下拉 z-50，弹窗遮罩 z-[100]，弹窗主体 z-[110]
- **理由**：解决实际开发中遇到的 bug — 搜索下拉和地址详情弹窗会同时出现，z-index 冲突导致弹窗被下拉遮挡。通过拉大层级差距确保弹窗始终在最上层

### 防止选择后二次搜索
- **决策**：使用 `justSelectedRef` 标志位阻止选择地址后触发重新搜索
- **理由**：用户从建议列表选择地址后，选中的地名会填入输入框，触发 value 变化 → 触发搜索。这不是用户意图，会导致刚关闭的下拉又弹出来。这是实际开发中发现的 bug，通过 Ref（而非 State）标志位解决，因为需要同步读写且不需要触发重渲染
